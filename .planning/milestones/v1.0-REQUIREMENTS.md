# Requirements: Hazn Platform

**Defined:** 2026-03-05
**Core Value:** Agents remember and compound -- every engagement builds on past decisions, brand voice, keyword history, and craft knowledge.

## v1 Requirements

### Infrastructure

- [x] **INFRA-01**: Platform runs via Docker Compose with Postgres, Letta, Vault, Redis, and Langfuse services
- [x] **INFRA-02**: Postgres 17 + pgvector schema supports all L2/L3 data (agencies, end_clients, keywords, audits, campaigns, decisions, approved_copy, brand_voice, vault_credentials)
- [x] **INFRA-03**: HashiCorp Vault stores all secrets; Postgres stores vault_secret_id references only
- [x] **INFRA-04**: Letta runs self-hosted via Docker with persistent storage

### Memory Layer

- [x] **MEM-01**: HaznMemory abstraction wraps all Letta access (swap-safe interface)
- [x] **MEM-02**: Agent can load L2+L3 client context at session start via load_client_context()
- [x] **MEM-03**: Context is injected into agent's active_client_context Letta block (<2s)
- [x] **MEM-04**: Checkpoint sync writes new learnings to Letta archival every 10 turns
- [x] **MEM-05**: Failure sync preserves partial learnings on crash (never discard)
- [x] **MEM-06**: Session end writes structured findings to Postgres and craft learnings to Letta archival
- [x] **MEM-07**: active_client_context block is wiped at session end (rebuilt fresh next session)
- [x] **MEM-08**: Agents can search their own archival memory semantically via search_memory()
- [x] **MEM-09**: Memory correction API allows programmatic override of incorrect memories
- [x] **MEM-10**: L3 context never bleeds between sessions (zero cross-client contamination)

### MCP Servers

- [x] **MCP-01**: mcp-hazn-memory server exposes load_context, write_finding, search_memory, checkpoint_sync, correct_memory, get_credentials tools
- [x] **MCP-02**: mcp-vercel server supports deploy, preview URL generation, domain management
- [x] **MCP-03**: mcp-github server supports repo management, PR creation, CI status
- [x] **MCP-04**: mcp-ga4 server supports GA4 data pull, GSC queries, benchmarks
- [x] **MCP-05**: mcp-pagespeed server supports Core Web Vitals and performance scoring

### Orchestrator

- [x] **ORCH-01**: Orchestrator loads L2+L3 context from Postgres at session start
- [x] **ORCH-02**: Orchestrator injects context into agent via mcp-hazn-memory
- [x] **ORCH-03**: Orchestrator manages session lifecycle (start, checkpoint every 10 turns, end/timeout at 4hr)
- [x] **ORCH-04**: Orchestrator writes workflow_runs to Postgres (metering source of truth)
- [x] **ORCH-05**: Orchestrator manages HITL queue and conflict flags
- [x] **ORCH-06**: Orchestrator interprets workflow YAML to coordinate multi-agent execution
- [x] **ORCH-07**: Orchestrator scopes tool access per agent type (principle of least privilege)

### Client Model

- [x] **CLT-01**: Three-layer client model: L1 (Autonomous), L2 (agencies), L3 (end-clients)
- [x] **CLT-02**: L2 agencies have house style, methodology, approved templates, tool preferences
- [x] **CLT-03**: L3 end-clients have brand voice, campaigns, keywords, competitors, history
- [x] **CLT-04**: L3 brand voice wins by default in L2/L3 conflicts
- [x] **CLT-05**: L2 can lock specific rules that override L3 (legal/compliance)
- [x] **CLT-06**: Agent flags L2/L3 conflicts in HITL queue with 24-hour timeout

### Credential Management

- [x] **CRED-01**: Orchestrator fetches secrets from Vault using vault_secret_id at runtime
- [x] **CRED-02**: Raw secrets never appear in agent context or LLM prompts
- [x] **CRED-03**: Credentials scoped per L2/L3 client (GA4, GSC, Ahrefs, CMS, Vercel, ESP)
- [x] **CRED-04**: get_credentials() MCP tool passes secrets directly to tool calls

### Vault Security

- [x] **VAULT-01**: Services authenticate to Vault via AppRole (not root token)
- [x] **VAULT-02**: Django AppRole has read+write access to secret/agencies/*
- [x] **VAULT-03**: Orchestrator AppRole has read-only access to secret/agencies/*
- [x] **VAULT-04**: MCP AppRole has read-only access to secret/agencies/*
- [x] **VAULT-05**: All AppRoles explicitly denied on sys/* and auth/* paths
- [x] **VAULT-06**: store_secret/read_secret helpers work transparently with AppRole auth
- [x] **VAULT-07**: get_credentials MCP tool works transparently with AppRole auth
- [x] **VAULT-08**: make vault-init creates AppRoles, policies, and writes .vault-approle
- [x] **VAULT-09**: Token caching avoids re-login; lazy re-auth handles expiry

### Observability & Metering

- [x] **OBS-01**: Langfuse SDK traces every LLM call with l2_client_id, l3_client_id, workflow_run_id tags
- [x] **OBS-02**: Postgres workflow_runs table tracks status, tokens, cost per workflow run (append-only)
- [x] **OBS-03**: workflow_agents table tracks per-agent turns, tokens, cost
- [x] **OBS-04**: workflow_tool_calls table tracks per-tool call count and cost
- [x] **OBS-05**: System flags runaway agents (>50 turns) and cost outliers (>$5/run)

### QA & Delivery

- [x] **QA-01**: QA Tester agent runs on every deliverable before marking done
- [x] **QA-02**: Production-ready criteria defined per task type (analytics, landing page, full site, blog, email, bug fix)
- [x] **QA-03**: Every approval gate has 48-hour timeout with default action
- [x] **QA-04**: Staging = Vercel preview URLs in v1

### Mode 3 Workspace

- [x] **WS-01**: Agency dashboard shows active projects, running workflows, deliverables
- [x] **WS-02**: Memory Inspector lets L2 agencies view and edit agent memory blocks
- [x] **WS-03**: End-Client Manager for creating/managing L3 profiles, brand docs, tool connections
- [x] **WS-04**: Workflow Trigger lets agencies run workflows from UI
- [x] **WS-05**: HITL Queue shows pending approvals across all end-clients
- [x] **WS-06**: Deliverables view for approving and sharing agent outputs

### Data Lifecycle

- [x] **DATA-01**: Maximum 90-day retention post-churn
- [x] **DATA-02**: GDPR on-request deletion within 30 days
- [x] **DATA-03**: L3 deletion is independent of L2 churn
- [x] **DATA-04**: Client notified at churn + 30 days before deletion

### Mode 2

- **MODE2-01**: MCP-connected access for technical founders via Claude Code

## Out of Scope

| Feature | Reason |
|---------|--------|
| Billing/subscription infrastructure | Blocked on Spike 6 cost data from 10+ real runs |
| Pricing model | Blocked on metering data |
| White-label/reseller | Not v1 |
| Mobile | Web-first |
| Real-time collaboration | Not needed for agent workflows |
| L3 client logins | Agencies control everything in v1 |
| Data migration for existing 10 agencies | Forward-only in v1 |
| Graph database | Postgres + pgvector covers ~80% of use cases |
| Multi-tenant auth (Mode 3) | Designed when Mode 3 UI is built |
| Memory quality control system | Known gap -- planned post-research phase |
