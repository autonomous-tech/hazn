# Roadmap: Hazn Platform

## Overview

Hazn transforms from stateless agent execution to a persistent, context-aware AI marketing platform. The journey starts with infrastructure (Postgres, Letta, Vault), builds the core memory abstraction that makes agents compound, layers orchestration and client model logic on top, adds external tool access and metering, validates everything with real internal engagements, then ships a self-serve agency workspace. Each phase delivers a verifiable capability that the next phase depends on.

### Phase 2: Memory Layer
**Goal**: Agents can load client context, accumulate learnings across sessions, and search their own memory -- all through a swap-safe abstraction
**Depends on**: Phase 1
**Requirements**: MEM-01, MEM-02, MEM-03, MEM-04, MEM-05, MEM-06, MEM-07, MEM-08, MEM-09, MEM-10, MCP-01
**Success Criteria** (what must be TRUE):
  1. HaznMemory.load_client_context() loads L2+L3 data from Postgres and injects it into the agent's active_client_context Letta block in under 2 seconds
  2. Checkpoint sync writes new learnings to Letta archival every 10 turns; failure sync preserves partial learnings on crash
  3. Session end writes structured findings to Postgres, craft learnings to Letta archival, and wipes active_client_context
  4. Agent can semantically search its own archival memory via search_memory() and receive relevant results
  5. Running two sequential sessions for different L3 clients produces zero cross-client data contamination (verified by inspection)
**Plans:** 3/3 plans complete

Plans:
- [ ] 02-01-PLAN.md -- HaznMemory types, MemoryCorrection model, core abstraction (load_client_context, search_memory, correct_memory)
- [ ] 02-02-PLAN.md -- Session lifecycle methods (checkpoint_sync, failure_sync, end_session) and integration tests for timing and isolation
- [ ] 02-03-PLAN.md -- mcp-hazn-memory FastMCP server with 6 tools (load_context, write_finding, search_memory, checkpoint_sync, correct_memory, get_credentials)

### Phase 3: Orchestrator & Client Model
**Goal**: The orchestrator manages complete workflow sessions -- loading context, coordinating agents, enforcing client hierarchy, handling credentials, and surfacing conflicts for human review
**Depends on**: Phase 2
**Requirements**: ORCH-01, ORCH-02, ORCH-03, ORCH-04, ORCH-05, ORCH-06, ORCH-07, CLT-01, CLT-02, CLT-03, CLT-04, CLT-05, CLT-06, CRED-01, CRED-02, CRED-03, CRED-04
**Success Criteria** (what must be TRUE):
  1. Orchestrator starts a session by loading L2+L3 context from Postgres, injecting it via mcp-hazn-memory, and writing a workflow_run record
  2. Orchestrator interprets workflow YAML to coordinate multi-agent execution with per-agent tool scoping (principle of least privilege)
  3. Session checkpoints every 10 turns, times out at 4 hours of inactivity, and writes final metering data on end
  4. L3 brand voice wins by default in L2/L3 conflicts; L2 locked rules override L3; unresolved conflicts enter HITL queue with 24-hour timeout
  5. Orchestrator fetches secrets from Vault at runtime -- raw secrets never appear in agent context or LLM prompts
**Plans:** 4 plans

- [ ] 03-02-PLAN.md -- Agent manager (persistent agents per type per client), session lifecycle, metering callback
- [ ] 03-03-PLAN.md -- L2/L3 conflict detection with LLM-based comparison, HITL queue state machine and timeout processing
- [ ] 03-04-PLAN.md -- DAG-based workflow executor with tool scoping, Celery async tasks, HITL polling API

### Phase 4: MCP Tool Servers & Observability
**Goal**: Agents can interact with external platforms (Vercel, GitHub, GA4, PageSpeed) and every workflow run is traced and metered for cost visibility
**Depends on**: Phase 3
**Requirements**: MCP-02, MCP-03, MCP-04, MCP-05, OBS-01, OBS-02, OBS-03, OBS-04, OBS-05
**Success Criteria** (what must be TRUE):
  1. mcp-vercel server can deploy a site, generate preview URLs, and manage domains
  2. mcp-github server can create repos, open PRs, and check CI status
  3. mcp-ga4 server can pull GA4 data, run GSC queries, and return benchmarks; mcp-pagespeed returns Core Web Vitals scores
  4. Every LLM call is traced in Langfuse with l2_client_id, l3_client_id, and workflow_run_id tags
  5. Postgres workflow_runs, workflow_agents, and workflow_tool_calls tables track tokens, cost, and turns per run -- system flags runaway agents (>50 turns) and cost outliers (>$5/run)

### Phase 7: Vault AppRole Authentication & Scoped Policies
**Goal:** Replace Vault root token authentication with production-grade AppRole authentication and least-privilege HCL policies so developers never see client secrets and each service gets only the Vault paths it needs
**Depends on:** Phase 1
**Requirements**: VAULT-01, VAULT-02, VAULT-03, VAULT-04, VAULT-05, VAULT-06, VAULT-07, VAULT-08, VAULT-09
**Success Criteria** (what must be TRUE):
  1. Three separate AppRole identities (Django, orchestrator, MCP) authenticate to Vault -- root token never used by application code
  2. Django AppRole can read+write at `secret/agencies/*`; orchestrator and MCP AppRoles are read-only
  3. All AppRoles are explicitly denied access to `sys/*` and `auth/*` paths
  4. HCL policy files are version-controlled in the repo and applied via `make vault-policies`
  5. `make vault-init` does full auto-setup: enables AppRole, creates roles, applies policies, generates Secret-IDs, writes `.vault-approle`
  6. Existing `store_secret`/`read_secret` helpers and `get_credentials` MCP tool work transparently with AppRole auth
  7. Token caching avoids re-login on consecutive calls; lazy re-auth handles token expiry
**Plans:** 2 plans
