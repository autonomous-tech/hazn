<objective>
Build the mcp-hazn-memory MCP server using FastMCP that exposes HaznMemory operations as tools callable by agents. The server maintains a _memory_registry so HaznMemory instances persist across tool calls within a session (checkpoint_sync actually has pending learnings to flush, write_finding does not trigger end_session). The server is a thin wrapper -- all logic lives in HaznMemory (Plan 02-01) and session lifecycle (Plan 02-02). This server is how agents interact with the memory system during orchestrated workflows.

Purpose: Agents need a standardized interface to load context, search memory, search cross-client insights, write findings, checkpoint, correct mistakes, and fetch credentials. MCP is that interface. The server delegates to HaznMemory, keeping the tool layer thin and the business logic centralized. The _memory_registry ensures stateful operations (add_learning -> checkpoint_sync) work correctly across separate tool calls.

Output: FastMCP server with 7 tools, unit tests, fastmcp dependency installed.
</objective>

    Create the FastMCP server:
    ```python
    import os
    import django
    os.environ.setdefault("DJANGO_SETTINGS_MODULE", "config.settings.local")
    django.setup()

    from fastmcp import FastMCP
    from hazn_platform.core.memory import HaznMemory
    from hazn_platform.core.memory_types import StructuredFinding, LearningSource
    from hazn_platform.core.vault import read_secret
    from hazn_platform.core.models import VaultCredential
    import uuid
    from datetime import datetime, timezone

    mcp = FastMCP("hazn-memory")

    # Session state: HaznMemory instances persist across tool calls
    # Keyed by agent_id so each agent gets its own stateful memory instance.
    # Full lifecycle (load/end) is the orchestrator's responsibility (Phase 3).
    # The registry just ensures stateful operations work within a session.
    _memory_registry: dict[str, HaznMemory] = {}

    def _get_or_create_memory(agent_id: str, l3_client_id: str, l2_agency_id: str) -> HaznMemory:
        if agent_id not in _memory_registry:
            _memory_registry[agent_id] = HaznMemory(
                agent_id=agent_id,
                l3_client_id=uuid.UUID(l3_client_id),
                l2_agency_id=uuid.UUID(l2_agency_id),
            )
        return _memory_registry[agent_id]
    ```

    Define 7 tools with `@mcp.tool()` decorator:

    1. `load_context(agent_id: str, l3_client_id: str, l2_agency_id: str) -> str`
       - "Load L2+L3 client context into agent's active memory block."
       - Gets or creates HaznMemory via _get_or_create_memory, calls load_client_context()

    2. `write_finding(agent_id: str, l3_client_id: str, l2_agency_id: str, finding_type: str, data: dict, workflow_run_id: str | None = None, agent_type: str = "unknown") -> str`
       - "Write a structured finding to Postgres without ending the session."
       - Gets or creates HaznMemory, creates StructuredFinding, calls memory.write_finding() (NOT end_session -- this is a standalone write)

    3. `search_memory(agent_id: str, l3_client_id: str, l2_agency_id: str, query: str, limit: int = 5) -> list[dict]`
       - "Search agent's archival memory semantically. Returns results ranked by similarity, recency, and confidence."
       - Gets or creates HaznMemory, calls search_memory()

    4. `search_cross_client_insights(agent_id: str, l3_client_id: str, l2_agency_id: str, query: str, finding_types: list[str] | None = None, limit: int = 10) -> list[dict]`
       - "Search structured findings from sibling L3 clients under the same agency. Requires agency cross_client_insights flag to be enabled."
       - Gets or creates HaznMemory, calls search_cross_client_insights()

    5. `checkpoint_sync(agent_id: str, l3_client_id: str, l2_agency_id: str) -> str`
       - "Flush pending learnings to archival memory."
       - Gets or creates HaznMemory, calls checkpoint_sync()

    6. `correct_memory(agent_id: str, l3_client_id: str, l2_agency_id: str, passage_id: str, new_content: str, reason: str, corrected_by: str) -> str`
       - "Soft-delete incorrect memory and create corrected replacement."
       - Gets or creates HaznMemory, calls correct_memory()

    7. `get_credentials(l3_client_id: str, service_name: str) -> dict`
       - "Fetch credentials for a service from Vault. Returns secret data for direct use in tool calls."
       - Queries VaultCredential.objects.get(end_client_id=UUID(l3_client_id), service_name=service_name)
       - Calls read_secret(credential.vault_secret_id)
       - Returns the secret dict

    IMPORTANT DESIGN NOTES:
    - The _memory_registry is server-side state. When the MCP server process runs, it persists HaznMemory instances across tool calls. This means checkpoint_sync will have actual _pending_learnings to flush (accumulated from prior add_learning calls via other tools or the orchestrator).
    - write_finding calls memory.write_finding() (standalone Postgres write), NOT end_session(). This is critical -- agents write individual findings during a session without triggering session cleanup.
    - Full session lifecycle management (load_context at start, end_session at end) is the Phase 3 orchestrator's responsibility. The MCP server provides the tools; the orchestrator calls them in the right order.
