  <action>
    Create `executor.py` with the DAG-based workflow execution engine:

    **class WorkflowExecutor:**
    - __init__(self, workflow: WorkflowSchema, session: WorkflowSession)

    - async run(self) -> WorkflowRun:
      1. Start session (session.start())
      2. Run pre-flight conflict detection: detect_conflicts(agency, end_client) -> process_conflicts(workflow_run, conflicts) -> session.log_conflicts(conflicts)
      3. Get execution waves via get_execution_order(workflow)
      4. For each wave:
         a. Execute all phases in the wave in parallel via asyncio.gather
         b. After each wave, check for checkpoints matching completed phases -> create HITL items
         c. If any blocking HITL items exist, set WorkflowRun status to "blocked" and return (workflow paused)
         d. Call session.checkpoint() after each wave
      5. After all waves complete: call session.end()
      6. Return final WorkflowRun

    - async _execute_phase(self, phase: WorkflowPhaseSchema) -> dict | None:
      1. Skip if phase has no agent (informational phases like "scope" in audit.yaml)
      2. Get or create agent: get_or_create_agent(phase.agent, l3_client_id, tool_ids=phase.tools)
      3. Reconcile tools: reconcile_tools(agent_id, phase.tools)
      4. Load context via session.get_memory(agent_id, end_client_id).load_client_context()
      5. Execute agent work (placeholder for now -- actual LLM interaction depends on Letta message API which will be wired in Phase 4)
      6. Store phase output as WorkflowPhaseOutput record
      7. Return output dict
      8. On exception: if phase.required -> re-raise (will halt in run()); if not required -> log and return None

    - _check_checkpoints(self, completed_phase_ids: set[str]):
      - For each checkpoint in workflow.checkpoints where checkpoint.after in completed_phase_ids:
        - create_hitl_item(workflow_run, "checkpoint", checkpoint.message, {"phase": checkpoint.after})

    Per CONTEXT.md decisions:
    - Parallel execution where dependencies allow
    - Phase outputs stored as database records (WorkflowPhaseOutput)
    - On phase failure: halt workflow, trigger failure_sync, flag as failed
    - Required phases block; optional phases can be skipped

    Per CONTEXT.md: tool access scoped per agent type via workflow YAML (ORCH-07). The executor calls reconcile_tools before each phase to enforce principle of least privilege.

    Note: The actual agent LLM interaction (sending messages to Letta agent and getting responses) is a stub in this plan. The executor sets up the correct agent, loads context, scopes tools, and manages the session lifecycle. Wiring to Letta's message API happens when the metering pipeline (Phase 4) provides the LLM callback hooks needed for cost tracking per RESEARCH.md Open Question 2.
