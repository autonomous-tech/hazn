      const { data: workflows, isLoading } = useQuery({

    The current code uses `useSSE([\`workflow-${runId}\`])` but the backend emits to `agency-{agency_id}` channels. Fix this per RESEARCH.md recommendation:
    - Get the agency_id from the auth context (useAuth hook or from the workflow run data)
    - Subscribe to the agency channel: `useSSE([\`agency-${agencyId}\`])`
    - The existing useSSE hook already invalidates `["workflows", event.run_id]` on workflow_status events, which will re-fetch the workflow run detail and update the chat timeline
    - If agencyId is not immediately available (loading), pass empty array to useSSE

    **4. Update WorkflowChat buildChatMessages to handle SSE-driven updates:**
    The existing buildChatMessages function builds messages from `run.phase_outputs`. This already works with SSE because:
    - SSE event arrives -> useSSE invalidates ["workflows", runId] -> TanStack Query refetches run detail -> buildChatMessages rebuilds timeline with new phase_outputs

    Add handling for failed phases:
    - Check `output.content?.status === "failed"` or the run's `error_details`
    - For failed phases, add an error message type:
      ```typescript
      type ChatMessage =
        | ... existing types ...
        | { type: "error"; phaseId: string; error: string; timestamp: string };
      ```
    - In buildChatMessages, if a phase output indicates failure (or if error_details mentions a phase), add an error ChatMessage
    - Add `ErrorBubble` sub-component: red-tinted card with AlertCircle icon showing the error text (per user decision: "Failed phases show as inline red error messages in WorkflowChat timeline")
    - Per user decision: "Completed phases still display their summaries even when workflow fails" -- the existing logic already handles this since phase_outputs are added independently before the status_change marker

    **5. Create deliverable-detail.tsx component:**
    ```typescript
    export function DeliverableDetail({ deliverableId }: { deliverableId: string }) {
      // Fetch deliverable detail
      const { data: deliverable } = useQuery({
        queryKey: ["deliverables", deliverableId],
        queryFn: () => api.get<Deliverable>(`/workspace/deliverables/${deliverableId}/`),
      });

      if (!deliverable?.html_content) {
        return <div className="text-muted-foreground">No rendered report available</div>;
      }

      return (
        <div className="space-y-4">
          <div className="flex items-center justify-between">
            <h3 className="text-lg font-medium">{deliverable.title}</h3>
            <Button variant="outline" size="sm" asChild>
              <a
                href={`/api/workspace/deliverables/${deliverableId}/html/`}
                target="_blank"
                rel="noopener noreferrer"
              >
                <ExternalLink className="mr-1 size-4" />
                Open Report
              </a>
            </Button>
          </div>
          {/* Inline iframe preview */}
          <iframe
            srcDoc={deliverable.html_content}
            className="h-[600px] w-full rounded-lg border"
            title={deliverable.title}
            sandbox="allow-same-origin"
          />
        </div>
      );
    }
    ```
    - The "Open Report" button links to the backend HTML view endpoint for full-page viewing
    - The iframe uses `srcDoc` to render the HTML inline without a separate request
    - `sandbox="allow-same-origin"` for security (no scripts in the report)

    **6. Update DeliverableCard in workflow-chat.tsx:**
    The existing DeliverableCard sub-component shows a basic card for deliverables. Update it to:
    - If the deliverable has html_content (check via the phase_output's content or linked deliverable), show a "View Report" button
    - Link to the deliverable detail page or open DeliverableDetail in a dialog

    **7. Ensure API client supports HTML endpoint:**
    If needed, add a method to lib/api.ts for fetching raw HTML (the existing api.get returns JSON). The deliverable-detail component uses `srcDoc` with JSON-fetched html_content field, so no raw HTML fetch is needed. The "Open Report" button uses a direct link.

    NOTE on SSE channel mismatch: The RESEARCH.md identifies this as Pitfall 5. The fix is straightforward: change the channel name from `workflow-${runId}` to `agency-${agencyId}` in WorkflowChat. The useSSE hook already handles event routing by invalidating query keys that include the run_id. The agencyId should come from the auth context (useAuth hook provides the current user's agency).
