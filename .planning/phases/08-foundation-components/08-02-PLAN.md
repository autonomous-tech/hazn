  <name>Task 2: Implement ToolRouter with static registry and dual-format dispatch (GREEN phase)</name>
  <files>hazn_platform/hazn_platform/orchestrator/tool_router.py</files>
  <action>
    Create `hazn_platform/hazn_platform/orchestrator/tool_router.py` following the design from 08-RESEARCH.md Pattern 2. Implementation details:

    1. **ToolRegistryEntry dataclass:**
       ```python
       @dataclass
       class ToolRegistryEntry:
           tool_name: str
           server_name: str
           description: str
           input_schema: dict[str, Any]
           callable: Any | None = None
       ```

    2. **ToolRouter class:**
       - `__init__`: Initialize `_registry: dict[str, ToolRegistryEntry]` (empty)
       - `register(entry)`: Add to registry by tool_name
       - `get_tool(name) -> ToolRegistryEntry | None`: Lookup by name
       - `list_tools() -> list[str]`: Return all registered tool names
       - `get_tools_for_server(server_name) -> list[ToolRegistryEntry]`: Filter by server
       - `to_anthropic_tools() -> list[dict]`: Export registry as Anthropic API tools array (for sending to the LLM). Each entry: `{"name": entry.tool_name, "description": entry.description, "input_schema": entry.input_schema}`. This is needed by Phase 9's AgentRunner.

       - `dispatch_anthropic(tool_use_block: dict) -> dict`:
         Extract id, name, input from the block. Look up in registry.
         On not found: return `{"type": "tool_result", "tool_use_id": id, "content": "Unknown tool: {name}", "is_error": True}`
         On found: try calling `entry.callable(**input)`, serialize result (json.dumps for dict/list, str for others).
         Return `{"type": "tool_result", "tool_use_id": id, "content": serialized_result}` (no is_error key on success).
         On exception: return `{"type": "tool_result", "tool_use_id": id, "content": "Tool error: {exc}", "is_error": True}`

       - `dispatch_agent_sdk(tool_name: str, tool_input: dict) -> dict`:
         Look up in registry.
         On not found: return `{"content": [{"type": "text", "text": "Unknown tool: {name}"}], "isError": True}`
         On found: try calling `entry.callable(**tool_input)`, serialize result.
         Return `{"content": [{"type": "text", "text": serialized_result}]}`
         On exception: return `{"content": [{"type": "text", "text": "Tool error: {exc}"}], "isError": True}`

    3. **build_tool_registry() -> ToolRouter:**
       Build the static registry. Per RESEARCH.md open question 1, use a **hardcoded static registry** mapping tool names to server names rather than relying on FastMCP internals (`_tool_manager`). This is more maintainable and the 4 servers / 20+ tools change infrequently.

       Implementation approach:
       - Define a `_STATIC_TOOL_MAP: dict[str, str]` at module level mapping every tool name to its server name. Derive this by reading the `@mcp.tool()` decorated functions from each of the 4 server files. The map should have ~20 entries.
       - `build_tool_registry()` creates a ToolRouter, iterates `_STATIC_TOOL_MAP`, and registers each entry with `callable=None` (callables are wired at runtime by Phase 9 when MCP servers are actually running).
       - Include `description` and `input_schema` for each tool based on the function docstrings and parameter signatures from the server files. For Phase 8, these can be minimal placeholder descriptions since the real tool schemas will be validated at integration time.
       - Add a `validate_registry()` function that attempts to import each MCP server module and compare registered tool names against actual `@mcp.tool()` registrations. This is called once at startup (Phase 9) as a sanity check. For Phase 8 tests, it is optional (mark test with `pytest.mark.integration`).

    4. **Logging:** Log at INFO level when tools are registered and when dispatch occurs. Log at WARNING for unknown tools and ERROR for tool exceptions.

    5. **No Django imports.** ToolRouter is a pure Python module. It does not import Django models, settings, or ORM. The `build_tool_registry()` function does NOT import MCP server modules (which have Django setup at top level). It uses the hardcoded static map.
